# Проект по проектированию базы данных обслуживания пассажиров в аэропорту
## Анализ предметной области
Предметная область данной работы охватывает процессы обслуживания пассажиров в аэропорту, начиная с момента оформления перевозки и заканчивая посадкой пассажира на воздушное судно. Обслуживание пассажиров представляет собой совокупность взаимосвязанных процедур, направленных на обеспечение безопасного, упорядоченного и своевременного перемещения пассажиров воздушным транспортом.

Аэропорт в рамках рассматриваемой предметной области выступает как инфраструктурный объект, обеспечивающий взаимодействие пассажиров, авиакомпаний и служб наземного обслуживания. Основное внимание уделяется не технической эксплуатации аэропорта, а процессам, непосредственно связанным с пассажиропотоком.

### Границы и допущения предметной области
В рамках данной работы предметная область ограничена процессами, непосредственно связанными с обслуживанием пассажиров в аэропорту.

В рамках системы учитываются:
- оформление билетов на рейсы;
- регистрация пассажиров;
- назначение мест в салоне воздушного судна;
- регистрация и учет багажа;
- посадка пассажиров на рейс.

Вне рамок системы находятся:
- техническая эксплуатация воздушных судов;
- управление экипажами;
- управление аэродромной инфраструктурой;
- финансовые расчеты и тарифная политика авиакомпаний;
- пограничный, таможенный и авиационный контроль.

### Основные бизнес-правила предметной области
В результате анализа были сформулированы следующие ключевые бизнес-правила:
- пассажир может иметь несколько билетов на разные рейсы;
- успешная регистрация приводит к формированию одного посадочного талона;
- билет может включать несколько дополнительных услуг;
- один билет может быть зарегистрирован не более одного раза;
- место в салоне может быть назначено только одному пассажиру на рейсе;
- вес багажа не может быть отрицательным;
- рейс имеет один аэропорт отправления и один аэропорт назначения;
- регистрация на рейс возможна только при наличии действительного билета;
- успешная регистрация приводит к формированию одного посадочного талона;
- каждому пассажиру при регистрации назначается одно место в салоне воздушного судна.

## Выделение сущностей и связей предметной области, построение концептуальной модели
На основе проведённого анализа предметной области обслуживания пассажиров в аэропорту был выполнен процесс идентификации ключевых сущностей и определения связей между ними. Данный этап соответствует концептуальному проектированию базы данных и направлен на формирование абстрактной модели предметной области, независимой от конкретной системы управления базами данных.

### Основные сущности предметной области
В результате анализа были выделены следующие основные сущности.

| Сущность  |  Описание |
| ------------ | ------------ |
| Аэропорт отправления и назначения  | Используется для представления пунктов отправления и назначения пассажирских рейсов. Необходимы в качестве справочных объектов, определяющих маршрут перевозки пассажиров  |
| Рейс  | Описывает запланированное выполнение пассажирской перевозки между аэропортами отправления и назначения в определённое время. Рейс используется как контекст, в рамках которого осуществляется обслуживание пассажиров  |
| Воздушное судно  | Представляет конкретное воздушное судно, используемое для выполнения пассажирских рейсов. Каждое воздушное судно может быть назначено на несколько рейсов при отсутствии временных пересечений  |
| Пассажирское место  | Описывает конкретное место в салоне воздушного судна, предназначенное для размещения пассажира на определённом рейсе. Пассажирское место принадлежит рейсу и может быть назначено пассажиру в процессе регистрации на рейс  |
| Пассажир  | Описывает физическое лицо, осуществляющее пассажирскую перевозку воздушным транспортом. Пассажир является участником процесса перевозки и может оформлять один или несколько билетов на различные рейсы  |
| Билет  | Описывает документ, подтверждающий право пассажира на перевозку на конкретном рейсе. Билет оформляется на одного пассажира и один рейс и может быть использован для прохождения регистрации, получения посадочного талона и дополнительных услуг  |
| Регистрация на рейс  | Описывает процесс подтверждения намерения пассажира совершить перелёт по билету на конкретном рейсе. Регистрация возможна только при наличии действительного билета и используется для оформления багажа и формирования посадочного талона  |
| Посадочный талон  | Описывает документ, формируемый в результате успешной регистрации на рейс и предоставляющий пассажиру право на посадку в воздушное судно. Посадочный талон содержит информацию, необходимую для организации процесса посадки  |
| Багаж  | Описывает багаж пассажира, передаваемый авиакомпании в процессе регистрации на рейс. Багаж оформляется в рамках регистрации и может включать ноль или несколько мест, связанных с конкретным пассажиром и рейсом  |
| Услуга  | Описывает факт предоставления дополнительной услуги в рамках конкретного билета. Сущность используется для фиксации связи между билетом и услугами, выбранными пассажиром  |

### Как спроектированная модель соответствует всем бизнес-правилам
| Бизнес-правило |  Подход к реализации | Описание |
|:----------------------------------------------:|:---------------------------------------------------------:|:----------------------------------------------------------------------:|
| Пассажир может иметь несколько билетов на разные рейсы                                              | Связь Passenger 1 : M Ticket через FK Ticket.passenger_id | Реализация хранения истории перелётов пассажира                        |
| Каждый билет относится только к одному рейсу                                                        | FK Ticket.flight_id                                       | Обеспечение принадлежности билета конкретному рейсу                    |
| Один билет может быть зарегистрирован не более одного раза                                          | UNIQUE (CheckIn.ticket_id)                                | Исключение повторной регистрации по одному билету                      |
| Успешная регистрация формирует один посадочный талон                                                | UNIQUE (BoardingPass.check_in_id)                         | Обеспечение взаимно-однозначной связи регистрации и посадочного талона |
| Место в салоне может быть назначено только одному билету                                            | UNIQUE (Ticket.seat_id)                                   | Предотвращение двойного назначения пассажирского места                 |
| В рамках одного рейса не может существовать двух одинаковых мест                                    | UNIQUE (flight_id, row_number, seat_letter)               | Обеспечение уникальности мест внутри рейса                             |
| Билет может включать несколько дополнительных услуг, а услуги могут использоваться в разных билетах | Ассоциативная сущность TicketService                      | Реализация связи многие-ко-многим                                      |
| Один и тот же билет не может содержать одинаковую услугу более одного раза                          | PRIMARY KEY (ticket_id, service_id)                       | Исключение дублирования услуг в одном билете                           |
| Рейс должен иметь аэропорт отправления и аэропорт назначения                                        | FK departure_airport_id и arrival_airport_id              | Реализация различных ролей сущности Airport                            |
| Плановое и фактическое время прибытия должно быть позже времени отправления                         | CHECK-ограничения в Flight                                | Контроль корректной временной последовательности рейса                 |
| Вес багажа, стоимость билета и стоимость услуги не могут быть отрицательными                        | CHECK (value > 0 / >= 0)                                  | Контроль допустимых значений числовых атрибутов                        |
| Регистрация на рейс возможна только при наличии действительного билета                              | BEFORE INSERT trigger ON CheckIn                          | Контроль бизнес-логики регистрации                                     |
| При регистрации пассажиру должно быть назначено место       | Проверка Ticket.seat_id в триггере регистрации            | Контроль обязательного распределения мест                              |
| Посадочный талон не может существовать без регистрации      | FK BoardingPass.check_in_id                               | Обеспечение ссылочной целостности                                      |
| Каждое место багажа должно относиться к конкретной регистрации | FK Baggage.check_in_id                             | Обеспечение связи багажа с процессом регистрации             |
| Большинство услуг не включается в тариф по умолчанию                                                | DEFAULT FALSE для included_in_fare                        | Автоматизация начального состояния услуги                              |

## Физическое проектирование базы данных на PostgreSQL
В качестве системы управления базами данных для реализации физической модели была выбрана PostgreSQL. Данная СУБД была выбрана благодаря поддержке реляционного подхода к проектированию баз данных, а также наличию встроенных механизмов обеспечения целостности данных. PostgreSQL поддерживает первичные и внешние ключи, ограничения CHECK, UNIQUE и DEFAULT, а также триггеры и пользовательские функции, что позволяет реализовать разработанные бизнес-правила непосредственно на уровне базы данных.

### Преобразование логической модели в физическую
В процессе физического проектирования были уточнены типы данных атрибутов, определены первичные и внешние ключи, настроены ограничения уникальности и реализованы проверки, обеспечивающие соблюдение бизнес-правил предметной области.
