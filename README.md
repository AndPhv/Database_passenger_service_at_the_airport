# Database_Airport_Operational
Проект по проектированию базы данных для информационной системы операционной деятельности аэропорта
## 1. Анализ предметной области
### 1.1. Общая характеристика предметной области
Аэропорт представляет собой сложный инфраструктурный объект, обеспечивающий приём, отправление и обслуживание воздушных судов. Операционная деятельность аэропорта включает в себя процессы планирования, координации и контроля выполнения авиарейсов, а также управление ресурсами, необходимыми для их обслуживания.

Эффективность операционной деятельности напрямую зависит от корректного и своевременного управления информацией о рейсах, воздушных судах, инфраструктуре аэропорта и экипажах. В связи с этим возникает необходимость в создании централизованной базы данных, обеспечивающей хранение и обработку соответствующих данных.

### 1.2. Цель и назначение проектируемой базы данных

Целью проектирования базы данных является создание информационной основы для поддержки операционной деятельности аэропорта, обеспечивающей:
- хранение структурированной информации о рейсах;
- учёт аэропортов отправления и назначения;
- планирование использования инфраструктуры аэропорта (терминалов и гейтов);
- назначение воздушных судов и экипажей на рейсы;
- контроль статусов рейсов и фактического времени их выполнения.

Проектируемая база данных ориентирована на использование в составе информационной системы аэропорта и может служить основой для последующего расширения функциональности.

### 1.3. Границы и допущения предметной области
В рамках данного проекта рассматривается операционная деятельность одного аэропорта, который далее будем называть базовым аэропортом.

В рамках системы учитываются:
- рейсы, обслуживаемые базовым аэропортом;
- аэропорты отправления и назначения рейсов;
- инфраструктурные объекты базового аэропорта (терминалы, гейты);
- воздушные суда, выполняющие рейсы;
- экипажи и их назначение на рейсы;
- статусы рейсов и временные параметры выполнения.

Вне рамок системы находятся:
- продажа и бронирование билетов;
- обслуживание пассажиров (регистрация, багаж, посадочные талоны);
- финансовые расчёты и биллинг;
- процессы авиационной безопасности и досмотра;
- техническое обслуживание воздушных судов.

Данные ограничения позволяют сосредоточиться на ключевых аспектах операционной деятельности и избежать избыточного усложнения модели.

### 1.4. Основные объекты предметной области
В ходе анализа предметной области были выделены следующие ключевые объекты.

#### 1.4.1 Аэропорты
Аэропорт рассматривается как географическая точка отправления или назначения рейса. В системе присутствует один базовый аэропорт, а также внешние аэропорты, используемые исключительно для указания маршрутов рейсов.

#### 1.4.2 Рейсы
Рейс является центральным объектом предметной области. Он характеризуется номером рейса, типом (прилёт или вылет), аэропортом отправления и назначения, плановыми и фактическими временными параметрами, а также текущим статусом выполнения.

Каждый рейс в рамках системы рассматривается как отдельное выполнение и не объединяется с регулярными или повторяющимися маршрутами.

#### 1.4.3 Инфраструктура аэропорта
Инфраструктура базового аэропорта включает:
- терминалы;
- гейты.

Терминалы служат логическим объединением гейтов, а гейты используются для непосредственного обслуживания рейсов. Назначение гейтов осуществляется с учётом временных ограничений, исключающих одновременное обслуживание нескольких рейсов одним и тем же гейтом.

#### 1.4.4 Воздушные суда
Воздушное судно представляет собой конкретный самолёт, выполняющий рейсы. Каждый самолёт относится к определённому типу, который характеризует модель и основные эксплуатационные характеристики.

Одно воздушное судно может выполнять несколько рейсов, при условии отсутствия временных пересечений.

#### 1.4.5 Экипажи
Экипаж состоит из членов лётного и обслуживающего персонала, назначаемых на конкретные рейсы. Для каждого назначения указывается роль члена экипажа в рамках выполнения рейса.

Система не хранит кадровую или персональную информацию, выходящую за рамки операционной деятельности.

#### 1.4.6 Статусы рейсов
Для контроля выполнения рейсов используется механизм статусов. Каждый рейс в каждый момент времени имеет один текущий статус, при этом история изменения статусов сохраняется для последующего анализа и аудита.

### 1.5. Основные бизнес-правила предметной области
В результате анализа были сформулированы следующие ключевые бизнес-правила:
- один рейс может быть обслужен только одним самолётом;
- одно воздушное судно не может быть назначено на пересекающиеся по времени рейсы;
- один гейт не может обслуживать более одного рейса одновременно;
- каждый рейс имеет ровно один текущий статус;
- история изменения статусов рейса должна сохраняться;
- член экипажа может быть назначен на несколько рейсов при отсутствии временных пересечений.

## 2. Выделение сущностей и связей предметной области, построение концептуальной модели
На основе проведённого анализа предметной области операционной деятельности аэропорта был выполнен процесс идентификации ключевых сущностей и определения связей между ними. Данный этап соответствует концептуальному проектированию базы данных и направлен на формирование абстрактной модели предметной области, независимой от конкретной системы управления базами данных.

### 2.1 Принципы выделения сущностей
Выделение сущностей осуществлялось на основе следующих принципов:
- каждая сущность отражает реальный объект или понятие предметной области;
- каждая сущность обладает устойчивым набором характеристик;
- между сущностями существуют логически обоснованные связи;
- сущности не дублируют функциональность друг друга.

### 2.2 Основные сущности предметной области
В результате анализа были выделены следующие основные сущности.

#### 2.2.1 Аэропорт - Airport
Сущность «Аэропорт» используется для представления географических точек отправления и назначения рейсов. В рамках проектируемой системы различаются:
- базовый аэропорт, для которого осуществляется управление операционной деятельностью;
- внешние аэропорты, используемые исключительно для указания маршрутов рейсов.

Данная сущность является справочной и используется в связях с сущностью «Рейс».

#### 2.2.2 Рейс - Flight
Сущность «Рейс» является центральной в проектируемой базе данных и отражает конкретное выполнение авиарейса. Рейс характеризуется номером, типом (прилёт или вылет), аэропортом отправления и назначения, а также временными параметрами выполнения.

#### 2.2.3 Тип рейса - FlightType
Сущность «Тип рейса» используется для классификации рейсов на прилётные и вылетные. Выделение данной сущности в отдельный справочник позволяет обеспечить целостность данных и упростить возможное расширение перечня типов рейсов в будущем.

#### 2.2.4 Статус рейса - FlightStatus
Сущность «Статус рейса» представляет собой справочник возможных состояний рейса в процессе его выполнения. Использование отдельной сущности для хранения статусов обеспечивает единообразие данных и возможность аналитической обработки.

#### 2.2.5 История статусов рейса - FlightStatusHistory
Для обеспечения контроля и анализа выполнения рейсов используется сущность «История статусов рейса», предназначенная для хранения информации о последовательности изменений статусов конкретного рейса во времени.

Выделение данной сущности позволяет:
- хранить текущее состояние рейса;
- анализировать задержки и отклонения от расписания;
- обеспечивать аудит изменений.

#### 2.2.6 Терминал - Terminal
Сущность «Терминал» отражает инфраструктурный объект базового аэропорта, предназначенный для обслуживания рейсов. Каждый терминал относится к одному аэропорту и объединяет набор гейтов.

#### 2.2.7 Гейт - Gate
Сущность «Гейт» представляет собой элемент инфраструктуры аэропорта, используемый для непосредственного обслуживания рейсов. Назначение гейта рейсу осуществляется с учётом временных ограничений, исключающих одновременное использование одного гейта несколькими рейсами.

#### 2.2.8 Воздушное судно - Aircraft
Сущность «Воздушное судно» описывает конкретный самолёт, выполняющий рейсы. Каждое воздушное судно относится к определённому типу и может быть назначено на несколько рейсов при отсутствии временных пересечений.

#### 2.2.9 Тип воздушного судна - AircraftType
Сущность «Тип воздушного судна» используется для классификации самолётов по модели и основным эксплуатационным характеристикам. Данная сущность является справочной и используется для унификации данных.

#### 2.2.10 Член экипажа - CrewMember
Сущность «Член экипажа» представляет персонал, участвующий в выполнении рейсов. В рамках системы хранятся только те данные, которые необходимы для операционной деятельности, без учёта кадровых и персональных аспектов.

#### 2.2.11 Роль экипажа - CrewRole
Сущность «Роль экипажа» используется для определения функциональной роли члена экипажа в рамках конкретного рейса (командир воздушного судна, второй пилот, бортпроводник).

#### 2.2.12 Назначение экипажа на рейс - CrewAssignment
Сущность «Назначение экипажа» служит для реализации связи многие-ко-многим между сущностями "Рейс" и "Член экипажа". Она позволяет фиксировать состав экипажа каждого рейса с указанием ролей.

### 2.3 Связи между сущностями
В результате выделения сущностей были определены следующие основные связи:
- рейс связан с аэропортами отправления и назначения;
- рейс обслуживается одним воздушным судном;
- рейс может быть назначен на один терминал и один гейт базового аэропорта;
- рейс имеет один текущий статус и историю изменения статусов;
- рейс обслуживается несколькими членами экипажа;
- член экипажа может участвовать в нескольких рейсах в разное время.

### 2.4 Построение концептуальной модели в нотации Питера Чена
На основе выделенных сущностей и связей предметной области была построена концептуальная модель базы данных операционной деятельности аэропорта. Для представления концептуальной модели использована нотация Питера Чена, которая является классической и широко применяется при концептуальном проектировании баз данных.
